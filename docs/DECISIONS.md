# 設計判断の記録（Decisions）

なぜ今の形になっているか。将来「変えていいか」を判断するための背景。新しい判断は追記する。

## 1. 単一 HTML ファイル・サーバーなし・ビルドなし
**判断**: 全体を 1 つの HTML に収め、`file://` で動くようにする。
**理由**: 「スキーマを AI に渡す→JSON 生成→即閲覧」を、環境構築ゼロで成立させるため。配布はファイルを渡すだけ。
**含意**: 依存を増やさない。外部 CDN もフォント以外は避ける。重い依存が要るなら別アプリ化を検討。

## 2. スタイルはインライン、デザイントークンを外部化しない
**判断**: CSS ファイル・クラス設計・トークン JS を作らず、`style` をインラインで書く。
**理由**: Design ランタイムの制約（クラス CSS はストリーミング描画を遅らせる）。加えて 1 ファイル完結を保つため。
**含意**: 値の一元管理は「テーマオブジェクト（STRUCT / RAMPS / PALETTES）」で行い、一覧は `DESIGN-SYSTEM.md` に文書化。色や寸法を変えるときはテーマ定義を直す。

## 3. データは外部 JSON、ビューアーに内蔵しない
**判断**: 実データは `assignees / calendar / projects / meta` を持つ JSON。ビューアーは読み取るだけ。
**理由**: AI がデータを生成し人が閲覧する、という関心の分離。表示ロジックとデータを疎結合に。
**含意**: `DATA-SCHEMA.md` が契約。破壊的変更は避け、`normalizeDoc` で後方互換を吸収。

## 4. 期間（日数）は画面側で計算、工数はデータ側
**判断**: タスクの「期間 = plan.start〜plan.end の暦日数」はビューアーが算出。工数 `personDays` はデータの値をそのまま。
**理由**: 期間は日付から一意に決まる派生値（二重管理でズレるのを防ぐ）。工数は稼働・見積りに依存する独立値。
**含意**: JSON に日数を持たせない。営業日ベース期間が必要なら別途オプション化。

## 5. タイムライン範囲にマイルストーンを含める
**判断**: `geo()` の範囲算出に、タスク予実だけでなくマイルストーン日付も含める。
**理由**: 最終タスクより後ろのマイルストーン（例: ストア公開）が枠外にはみ出すバグがあったため。
**含意**: 描画範囲に効く日付源を増やすときは `geo()` の走査に必ず追加する。

## 6. 稼働日はデータ側で設定（祝日を内蔵しない）
**判断**: 祝日 API を内蔵せず、`calendar.workweek` / `holidays` / `extraWorkdays` を利用者（＝生成する AI）が指定。
**理由**: 祝日は国・年・企業カレンダーで異なり、内蔵すると陳腐化・不正確になる。土日稼働や休日出勤も表現したい。
**含意**: 判定優先順位は 特別稼働日 > 休業日 > 週末。ビューアーは与えられたカレンダーを忠実に描くだけ。

## 7. 網掛けは行背景を半透明にして透けさせる
**判断**: プロジェクト行・交互行の背景を不透明にせず、半透明にする（左固定セルのみ不透明）。
**理由**: 不透明背景が背後の週末/祝日網掛けを隠し、網掛けが「まだら」に見えるバグがあったため。
**含意**: 行背景色を足すときは半透明（`hex(color, a)`）を使う。左セルはスクロールするバーを隠すため不透明を維持。

## 8. バー注記はバーの「外」に置く（内側に文字を入れない）
**判断**: 担当者＝バー左外、タスク名＝バー右外、日付＝左右外。バー内にテキストを置かない。
**理由**: バー内は進捗塗り（濃色）と未塗り（淡色）が混在し、どの文字色でも一部が読めなくなる。短いバーには入らない。
**含意**: 担当者はアバター（自前で色を持つチップ）なので視認性が背景に依存しない。名前は左ツリーにもあるので右添えで十分。

## 9. ホバーは自前ツールチップ（標準 title に頼らない）
**判断**: `title` 属性ではなく、`state.tip` ＋ `renderTip()` の自前カードで即時表示。
**理由**: 標準 title は表示が遅く、環境によっては出ないため「情報が無い」ように見えた。
**含意**: バーのホバー領域は `onMouseEnter/Leave` で `showTip/hideTip`。カードは `pointerEvents:none` でちらつきを防ぐ。

## 10. テーマは「構造 × 配色」の 2 軸
**判断**: レイアウト/密度/明暗（A/B/C）と配色（11 パレット）を直交させ、`buildTheme` で合成。
**理由**: 「デンスなダーク × テラコッタ」のような自由な組み合わせを、少ない定義で成立させるため。
**含意**: 新しい見た目は、構造か配色のどちらの軸に足すかを決めてから追加する。

## 11. 担当者 `who` は人名に限定しない
**判断**: `who` に会社コード（`CO` / `XYZ` 等）も許容。色・イニシャルは未定義でも自動生成。
**理由**: 小規模プロジェクトではタスクの主体が個人でなく会社単位になることがある。
**含意**: アバターはイニシャル最大 3 文字まで表示。`assignees` 未定義の `who` でも破綻しない。
